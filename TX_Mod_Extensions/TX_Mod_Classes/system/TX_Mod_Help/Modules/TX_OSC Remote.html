OSC Remote


Module type: Action


Introduction:


This module is designed for linking up an external device that sends OSC messages - such as a phone or an iPad running TouchOSC, or another computer running the TX Modular system or other OSC software.

It can learn the IP Address and all the different OSC messages sent by the remote device.

Each OSC String is linked to an Action within the TX. For more detials on Actions, see help file What_Are_Actions

There's a button to sync the remote device with the TX modular by sending it OSC messages. Sync messages will automatically be sent whenever a saved system is opened, and when system snapshots are loaded. (You can also set it up so that, by clicking a button on your remote device, an OSC message triggers an action to sync the remote device.)

Actions are triggered when OSC messages are received as long as the OSC message begins with a string that matches the OSC string on the screen. The string matching is case sensitive - upper and lower case, G versus g, are treated as different letters.

The OSC message sent should consist of the OSC string followed by any arguments, which can be numbers or strings (e.g. 9, 4.77, "on").

OSC message arguments can be passed to the Actions.

For further details about OSC see help file: OSC_Info


Controls:


Remote Port - this is the port on the remote OSC device that messages will be sent to when syncing.

IP Address - this is the IP Address of the remote OSC device that messages will be sent to when syncing.

OSC Learn - automatically detect IP Address - when switched on, this will wait for the next OSC message received and assign the IP Address automatically.

Sync Now - send stored values for active OSC strings - this will send OSC messages to the remote device to update the current values using all OSC strings which are switched on and have "use args" selected.

OSC Learn - automatically detect OSC strings - when switched on, this will listen for OSC messages received, create String and Action rows and assign the strings automatically.

Note: the following OSC strings are used internally by SuperCollider and so shouldn't be used for OSC communication. They will be ignored by OSC Learn:

/status.reply, /tr, /done, /synced, /n_on, /n_off, /n_move, /n_end, /n_go.


String and Action Rows - the following controls are on each row:

[] yellow buttons to set row highlight on or off, where a yellow box appears behind the row and the action parameters are larger. When a row is edited, it will automatically be highlighted.

+/- buttons - these are for adding or deleting a row of controls. When you add a row, the previous row's OSC string is copied.

OSC string (sorted alphabetically) - this string will be checked against the first item in any OSC messages received. If the string matches, the Actions will be triggered.

on- off - this turns the action on or off.

trigger type - the following choices are given: "Always trigger", "Trigger when arg = 1", and "Trigger when arg = 0". The last 2 are useful for example with on/off button OSC messages. These options allow you to run actions specifically when a value is turned ON or OFF - 1 meaning "ON", and 0 meaning "OFF".

use args - if switched on, this means that any arguments that come with the OSC string will be put into the value settings for the Action.

The arguments can either be numbers or text strings.

For example, if the Action selected was to set the frequency of a filter, or the BPM of a sequencer, then a number could be passed to set the value.

If the Action selected had a checkbox to change settings (such as the Reverse checkbox in a Sample Player module), the OSC argument can be: (a) a text string with "ON" or "OFF", or (b) a number with 1 meaning "ON", and 0 meaning "OFF".

If the Action selected had a popup to change settings (such as the type of Waveform in a Wave Synth module), the OSC argument can be: (a) a text string of the popup value, or (b) a number representing the index of the popup. (N.B. Index numbers start from 0, so 0 would be the index of the first item in the popup, 1 would be the index of the second item, 2 would be the index of the third item, etc.)

from no - this refers to the first argument that will be used for value settings. By default this will be 1, meaning the first OSC argument after the OSC string. Sometimes where OSC messages contain several arguments, higher values may be needed. For example, when using 2D sliders in TouchOSC iPad software, the X and Y values are sent as arguments 1 & 2. In this case, you might to create 2 rows, each with the same OSC string, but using different arguments numbers to run 2 different actions - for example controlling filter frequency and filter resonance.

module - selects which module will be used.

action - selects Action will be run out of all the possible Actions for the selected module.

value settings - depending on which Action is run, values can be set for the Action.

For example, if the Action selected was to set the Frequency of a Waveform module, then a slider would be displayed for setting the frequency.
